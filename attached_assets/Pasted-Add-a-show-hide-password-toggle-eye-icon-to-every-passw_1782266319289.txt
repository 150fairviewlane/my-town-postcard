Add a show/hide password toggle (eye icon) to every password input field on the dealer-facing auth pages: dealer login, dealer signup (where a password is first set), and the password-reset page. Default to masked (current behavior), with a toggle that reveals plain text on demand — this is standard modern UX and doesn't reduce security meaningfully, since the actual protection masking provides is against someone physically looking at the screen, which a user-controlled toggle still allows for.

1. For each password `<input>` field, add a small eye/eye-slash icon button positioned inside the input (typically right-aligned) that toggles the input's `type` attribute between `"password"` and `"text"` on click.

2. Use a simple icon library already available in the project (check for `lucide-react` or similar already in use elsewhere) for the eye/eye-slash icons — don't add a new dependency for this.

3. The toggle should be a plain button (not a submit button) so clicking it never triggers form submission — use `type="button"` explicitly.

4. **Preserve the correct `autocomplete` attribute on each field regardless of toggle state** — `autocomplete="current-password"` on the login page's password field, `autocomplete="new-password"` on the signup and password-reset pages' password fields. Confirm these attributes are already present; if not, add them (this is good practice independent of the visibility toggle and helps browsers autofill/save passwords correctly).

5. Style the toggle icon to match existing input styling — small, muted gray by default, slightly darker on hover, sized to fit cleanly within the existing input field's height without changing the input's overall dimensions.

6. Apply this to all three pages: dealer login, dealer signup (password field), and password reset (new password field) — same toggle behavior and styling on each for consistency.

After implementing, test: toggling visibility shows/hides the typed password correctly without losing the typed value, the toggle button never submits the form, and password managers (browser-native or third-party) still correctly offer to save/fill the password given the `autocomplete` attributes.
