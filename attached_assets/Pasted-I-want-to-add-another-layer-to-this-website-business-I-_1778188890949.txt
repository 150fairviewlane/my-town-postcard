Territory Assignment — The Real Challenge
The “4 distinct postcard areas” requirement is genuinely hard. Here’s why:
A postcard area = ~5,000 homes for EDDM. That’s roughly 2-4 USPS carrier routes depending on density. So each dealer territory needs ~20,000 households worth of contiguous-but-distinct geographic clusters.
Free APIs that can help:
	1.	Zippopotam.us — Free, no key needed. Returns lat/lng for any ZIP. Good for distance calculations.
	2.	US Census Geocoder — Free, official. Can return demographic data per ZIP.
	3.	HUD USPS ZIP-County Crosswalk — Free download, maps ZIPs to counties.
My recommended approach — a hybrid algorithm using free data:
	1.	Dealer enters their home ZIP
	2.	We pull lat/lng from Zippopotam.us (free)
	3.	We calculate all ZIPs within ~25 miles using a pre-loaded ZIP database (free, ~42,000 ZIPs total — small enough to ship with the app)
	4.	We cluster those nearby ZIPs into 4 groups using K-means clustering (free, runs in JS)
	5.	We filter clusters to only include ZIPs with sufficient household density (need household data)
	6.	Show the dealer their proposed 4 territories on a map for confirmation
The household density piece is the tricky one. HUD’s free crosswalk gives us ZIP+county, and the Census API gives population per ZIP — combined, we can estimate households (population ÷ 2.5).
I’ll bundle a static JSON file with ZIP→{lat, lng, population, county, state} for all US ZIPs (~3MB, perfectly fine) so we don’t depend on any external API at runtime. This is bulletproof.
