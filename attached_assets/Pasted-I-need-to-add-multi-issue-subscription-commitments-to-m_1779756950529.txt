## Stripe Webhooks (Critical Infrastructure — Build This Carefully)

Webhooks are the source of truth for subscription state. Without them, the local database will drift out of sync with Stripe and lead to billing errors, missed cancellations, or printed ads for customers who didn’t actually pay. Build this as core infrastructure, not as an afterthought.

### Check existing webhook setup first

Before building new webhook handlers, check whether the current app already has a Stripe webhook endpoint. If it does, confirm whether it verifies the Stripe signature (look for stripe.webhooks.constructEvent or equivalent). If signature verification is missing, add it before doing anything else — an unverified webhook endpoint is a security hole that lets anyone send fake events to corrupt the database.

### Required webhook endpoint

Create or update a single webhook endpoint (e.g., POST /api/webhooks/stripe) that:

1. Verifies the Stripe signature using the STRIPE_WEBHOOK_SECRET environment variable (store this in Replit Secrets — one for test mode, one for live mode). Reject any request that fails signature verification with HTTP 400. Use stripe.webhooks.constructEvent with the raw request body, the stripe-signature header, and the webhook secret.
1. Returns 200 OK to Stripe within 5 seconds, even on processing errors. If processing takes longer or fails, log the event for retry but still acknowledge receipt to Stripe.
1. Is idempotent — Stripe retries webhooks on failure, sometimes delivering the same event multiple times. Create a stripe_webhook_events table (event_id primary key, processed_at timestamp) and check if the event ID has already been processed before acting on it.

### Events that must be handled

For each of the following, update the local database accordingly:

- customer.subscription.created — Confirm subscription exists in local DB, mark active, set commitment_start_date and commitment_end_date from Stripe data.
- customer.subscription.updated — Update local subscription record. Pay special attention to status changes (active → past_due, active → canceled, etc.) and cancel_at changes.
- customer.subscription.deleted — Mark subscription as ended in local DB. Stop including this advertiser in future issues. Send me an admin notification.
- invoice.payment_succeeded — Record successful payment, decrement issues_remaining if appropriate, ensure advertiser stays active for next issue.
- invoice.payment_failed — Mark subscription as past_due in local DB. Send advertiser a payment retry email (Stripe handles dunning automatically, but log the event). After Stripe’s final retry fails and subscription transitions to canceled or unpaid, the subscription.updated or subscription.deleted webhook will fire — handle removal from issue lineup there.
- invoice.upcoming — Fires ~7 days before next invoice. Use this to send the advertiser a heads-up email, especially before final invoice of their commitment so they can renew.
- charge.refunded — If a payment is refunded, update payment status in local DB and flag for admin review.

### Failure modes to handle

1. Webhook arrives before the checkout success redirect. Don’t assume the local DB record exists when the webhook arrives — use upsert logic (create if missing, update if present) keyed on Stripe customer ID and subscription ID.
1. Webhook never arrives. Build an admin “Sync with Stripe” button that pulls current subscription state from Stripe API and reconciles with local DB. Useful for debugging and as a manual recovery option.
1. Out-of-order delivery. Use the event’s created timestamp to detect stale updates — if an incoming event is older than the last update on that subscription, log it but don’t overwrite newer data.

### Webhook logging and monitoring

Every webhook event should be logged with: event ID, event type, timestamp received, processing status (success/failed/skipped), and any error message. Build a simple admin view showing the last 100 webhook events so I can debug issues without diving into Stripe’s dashboard.

### Webhook testing

Use the Stripe CLI to forward webhook events to localhost during development: stripe listen –forward-to localhost:PORT/api/webhooks/stripe. Test each event type using stripe trigger <event_name> before deploying.

### Production webhook registration

After deploying, register the production webhook endpoint in the Stripe Dashboard under Developers → Webhooks. Subscribe only to the events listed above (not all events) to reduce noise. Confirm the webhook secret is set as an environment variable in production.

## Edge Cases to Handle

- What happens if an advertiser’s payment fails mid-commitment? Use Stripe’s smart retries and dunning emails. The invoice.payment_failed webhook marks them past_due. During the retry window (typically 2 weeks), do NOT include their ad in the next issue if their status is past_due. If they recover, status returns to active and they resume in the next issue. If they fail permanently, the subscription.deleted webhook removes them.
- What if I need to refund or cancel early? Build an admin “cancel subscription” action that uses Stripe’s cancel_at_period_end or immediate cancellation with prorated refund.
- What if an advertiser wants to change their ad design between issues? They should be able to log back into their account, edit the ad (regenerate via Grok or upload new content), and the new version applies to the next unprinted issue.
