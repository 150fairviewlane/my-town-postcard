Rebuild the territory builder from scratch using a new city-hub model.
This replaces the entire Census CBP county-based approach with a simpler,
more accurate system based on city commercial hubs.

Do not modify Georgia territory data, the MANAGED_STATES guard, the
territory finder map, the admin territory manager, or any other feature.
This change is isolated to the territory builder engine and Census API
module only.


════════════════════════════════════════════════
OVERVIEW — THE NEW MODEL
════════════════════════════════════════════════

A territory = 4 commercial hub cities, each qualifying independently,
forming a geographic cluster around the dealer's ZIP code.

A city QUALIFIES as a hub if:
  households ≥ 5,000  (sum of all ZIP households within 15 miles)
  businesses ≥ 100    (sum of all ZBP establishments within 10 miles)

A territory is VALID if:
  It has exactly 4 qualifying hub cities
  All 4 are within 40 miles of the dealer's ZIP centroid
  The 4 cities spread in reasonably different directions
  (not all clustered in one quadrant)

This model works identically in all 50 states with no state-specific
exceptions, no county adjacency logic, no fringe shifts, and no
household proxy math.

── STEP B: FIND CANDIDATE HUB CITIES ──────────

  async function findCandidateHubs(
    lat: number,
    lng: number
  ): Promise<CityHub[]>

  interface CityHub {
    cityName: string;
    stateName: string;
    stateAbbr: string;
    lat: number;
    lng: number;
    catchmentHouseholds: number;
    nearbyBusinesses: number;
    distanceFromDealer: number;
    qualifies: boolean;
  }

  Algorithm:
  1. Load all cities from the Gazetteer Places file (already loaded)
     that are within TERRITORY_SEARCH_RADIUS (40 miles) of (lat,lng)
     Filter to same state as the dealer ZIP

  2. For each candidate city:
     a. Sum households: call getZipsNearLocation(city.lat, city.lng,
        HUB_HOUSEHOLD_RADIUS) and sum households field
     b. Sum businesses: call getZipsNearLocation(city.lat, city.lng,
        HUB_BUSINESS_RADIUS) and sum businesses field
     c. Set qualifies = households >= HUB_MIN_HOUSEHOLDS
                     AND businesses >= HUB_MIN_BUSINESSES
     d. Calculate distance from dealer ZIP centroid

  3. Return all candidates sorted by:
     - qualifies = true first
     - then by distanceFromDealer ascending

  PERFORMANCE NOTE: The Gazetteer file has ~30,000 US places.
  Filtering to within 40 miles before computing catchment areas
  is critical for performance. Use a bounding box pre-filter:
    lat ± 0.6 degrees (~40 miles)
    lng ± 0.8 degrees (~40 miles at mid-latitudes)
  Then apply precise Haversine check only on the ~50-200 cities
  that pass the bounding box filter.


── STEP C: SELECT 4 BEST HUBS ──────────────────

  function selectBestHubs(
    candidates: CityHub[],
    dealerLat: number,
    dealerLng: number
  ): CityHub[]

  From the qualified candidates, select TARGET_HUB_COUNT (4) hubs
  that form a good geographic spread.

  Algorithm:
  1. Filter to only qualifying candidates
     (households >= 5000 AND businesses >= 100)

  2. If fewer than MIN_HUB_COUNT (3) qualify:
     Return all qualifying candidates (territory marked not_viable)

  3. If 3 or 4 qualify: return them all

  4. If more than 4 qualify, select the best 4 using
     directional spread:

     a. Always include the closest qualifying city (anchor)

     b. Divide the remaining candidates into 4 quadrants
        relative to the dealer ZIP:
          NE: lat > dealer_lat AND lng > dealer_lng
          NW: lat > dealer_lat AND lng < dealer_lng
          SE: lat < dealer_lat AND lng > dealer_lng
          SW: lat < dealer_lat AND lng < dealer_lng

     c. Pick the closest qualifying city from each quadrant
        that hasn't been selected yet

     d. If a quadrant has no qualifying city, pick the next
        closest qualifying city overall

  This ensures the 4 hubs surround the dealer ZIP rather than
  clustering in one direction.


── STEP D: BUILD THE PROPOSAL ──────────────────

  async function buildCityHubProposal(
    zip: string,
    lat: number,
    lng: number,
    stateAbbr: string
  ): Promise<TerritoryProposal>

  interface TerritoryProposal {
    proposedName: string;
    slug: string;
    stateAbbr: string;
    stateName: string;
    hubs: CityHub[];           // the 4 selected hub cities
    totalHouseholds: number;   // sum of all hub catchment households
    totalBusinesses: number;   // sum of all hub nearby businesses
    isViable: boolean;         // true if >= MIN_HUB_COUNT hubs qualify
    hubCount: number;          // actual number of qualifying hubs
    centroidLat: number;       // average lat of all hubs
    centroidLng: number;       // average lng of all hubs
    viabilityMessage: string;  // human-readable assessment
  }

  Steps:
  1. findCandidateHubs(lat, lng)
  2. selectBestHubs(candidates, lat, lng)
  3. Build name from hub city names:
       1 hub:  "{City} Area"
       2 hubs: "{City1} / {City2}"
       3 hubs: "{City1} / {City2} / {City3}"
       4 hubs: "{City1} / {City2} / {City3} / {City4}"
       If name > 50 chars: "{City1} / {City2} / {City3} Area"
  4. Generate slug (same logic as before)
  5. Calculate centroid as average lat/lng of all hub cities
  6. Set viabilityMessage:
       totalBusinesses >= 600: "Strong market — excellent pipeline
         for 4+ postcard mailings."
       totalBusinesses 400-599: "Good market — solid pipeline for
         4 postcard mailings."
       totalBusinesses 300-399: "Adequate market — manageable for
         4 mailings with focused sales effort."
       totalBusinesses < 300: "⚠ Smaller market — this territory
         may take longer to fill postcard spots. You'll have a
         larger exclusive area."
       hubCount < 4: "⚠ Only {hubCount} mailing areas found near
         this location. Contact us to discuss territory options."
