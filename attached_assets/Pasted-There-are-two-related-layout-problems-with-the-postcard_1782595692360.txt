There are two related layout problems with the postcard ad picker on iPhone (landscape only — the existing forced-landscape "rotate your device" prompt for portrait phones under 768px wide is intentional and must NOT be changed or removed), in artifacts/localspot/src/PostcardPickerSection.jsx and the sticky NavBar in artifacts/localspot/src/landingSections.tsx:

1. SCROLL BEHAVIOR: The NavBar (landingSections.tsx, ~line 74) is `position: sticky, top: 0` and stays pinned permanently, even while the picker section is in view. It overlaps the top of the picker section and effectively steals that much vertical space from it. I want the NavBar to remain sticky while the user is above the picker section, but once the user scrolls down far enough that the picker section (.postcard-section in PostcardPickerSection.jsx) is about to enter view, the NavBar should stop being sticky and scroll away normally, so the picker section gets the full screen height with nothing overlapping it. Implement this with an IntersectionObserver watching the picker section's top edge, toggling the NavBar's position between "sticky" and "static" (or an equivalent CSS class) based on whether the picker section is approaching/in view.

2. SIZING: In PostcardPickerSection.jsx (~line 728-759), the postcard-section container is `height: 100vh`, and the postcard's size is computed with a hardcoded guess: `maxWidth: "calc((100vh - 160px) * 12 / 9)"`. This is wrong on two counts:
   - "160px" is a hardcoded guess for the header/controls height above the postcard, not a real measurement, so it's wrong whenever that header content's actual rendered height differs.
   - It only solves for the postcard's width assuming height is the binding constraint — it never checks if the available width is actually smaller. On iPhone landscape (wide but short), width can easily end up being the tighter constraint, so the postcard renders smaller than the truly available space, leaving empty white space below it.

   Fix: make the postcard size dynamically to fill whatever space is actually left over after the header above it (and after the NavBar has scrolled away per #1), using BOTH the available width and available height — pick the largest size that maintains the 12:9 aspect ratio without exceeding either dimension (like CSS object-fit: contain, but for sizing a box, not an image). Use a ResizeObserver or the existing ref-based measurement in this file to measure real available space at render time, instead of a hardcoded subtraction. Use `dvh` instead of `vh` for the outer container height if not already, to avoid the iOS Safari toolbar viewport issue.

   Goal: every ad spot stays fully visible, the postcard fills as much of the actual available screen as it can without being cropped, and there's no leftover blank space below it.

Do not touch the .rotate-prompt / .postcard-section media query logic that forces landscape on small portrait screens — that behavior is intentional and should be preserved exactly as-is.

Test on iPhone in landscape orientation across a couple of screen sizes (e.g. iPhone SE and iPhone 14/15 Pro Max) to confirm the postcard fills the available space correctly on both a short/narrow landscape viewport and a wider one.
