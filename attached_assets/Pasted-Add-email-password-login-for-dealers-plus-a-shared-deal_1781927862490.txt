Add email/password login for dealers, plus a shared "dealer territory overview" page usable by both admins (viewing any dealer) and dealers themselves (viewing only their own data). This replaces the plan to duplicate the admin dealer-detail page — instead, build ONE shared display component fed by two different auth-gated data sources.

**Context:** `dealersTable` currently has NO password column — the schema comment explicitly says "dealer login + dashboard come in later phases." This is genuinely new functionality, not a tweak to existing auth. The existing `portalToken`-based `/my-territory?token=...` link-based portal can stay as-is for now (don't remove it) — we're adding a real login ON TOP, not replacing that yet.

---

## Part 1: Schema — add password to dealers

In `lib/db/src/schema/dealers.ts`, add a `passwordHash` column (text, nullable initially since existing dealers won't have one yet):

```ts
passwordHash: text("password_hash"),
```

Generate and run the migration.

---

## Part 2: Password hashing

Install `bcrypt` (or `bcryptjs` if native bcrypt has issues in this environment) if not already a dependency. Create a small helper module (e.g. `artifacts/api-server/src/lib/dealerAuth.ts`) with:

```ts
async function hashPassword(plain: string): Promise<string>
async function verifyPassword(plain: string, hash: string): Promise<boolean>
```

Use bcrypt with a reasonable cost factor (10-12 rounds). NEVER store or log plaintext passwords anywhere, including in request logs — make sure any request logging middleware doesn't accidentally log the raw `req.body` for password-containing endpoints.

---

## Part 3: Set password at dealer creation (both paths)

1. **Public dealer signup flow** (`POST /api/dealers` or wherever the public signup form submits): add a required `password` field to the request schema (Zod). Hash it and store in `passwordHash` when creating the dealer row.

2. **Admin "create dealer" flow** (wherever admin manually creates a dealer — check the admin dealers page / custom territory tool's dealer-creation step): add a `password` input field to that form too. Same hashing, same storage.

Both paths converge on the same `passwordHash` column — login works identically regardless of how the dealer was created.

---

## Part 4: Dealer login endpoint

Add `POST /api/dealer/login`:

```json
{ "email": "dealer@example.com", "password": "..." }
```

1. Look up dealer by email (case-insensitive, matching the existing `dealersTable` unique email index).
2. If not found, or `passwordHash` is null (e.g. an old dealer created before this feature), or password doesn't match → return a generic `401 { error: "Invalid email or password." }` (don't reveal which part was wrong — standard practice to avoid email enumeration).
3. On success, issue a JWT (mirror the existing admin JWT pattern in `routes/dealers.ts`/`routes/admin.ts` — same `jsonwebtoken` library, same `SESSION_SECRET`) with payload `{ dealerId: dealer.id }`, expiring in 7 days (longer-lived than the 24h admin token since dealers will check this less frequently — use your judgement, 7 days is a reasonable default).
4. Return `{ token, dealerId, name: dealer.name }`.

Add a `requireDealerAuth` middleware (mirrors `requireAdmin`) that verifies this JWT and attaches `req.dealerId` for use by protected dealer routes.

## Part 5: Forgot password flow

1. `POST /api/dealer/forgot-password` — body: `{ "email": "..." }`. Look up the dealer; if found, generate a random reset token (UUID, similar to the existing `portalToken` pattern) and store it with an expiry (add `passwordResetToken` (uuid, nullable) and `passwordResetExpiresAt` (timestamp, nullable) columns to `dealersTable`). Send a reset email via the existing Resend setup (`lib/emails.ts` — add a new `sendDealerPasswordResetEmail` function following the same pattern/branded-footer as existing email functions) containing a link like `https://mytownpostcard.com/dealer/reset-password?token=<token>`. Token expires in 1 hour. ALWAYS return a generic success message regardless of whether the email was found ("If an account exists for this email, a reset link has been sent.") — don't reveal whether an email is registered.

2. `POST /api/dealer/reset-password` — body: `{ "token": "...", "newPassword": "..." }`. Validate the token exists, matches a dealer, and hasn't expired. Hash the new password, update `passwordHash`, clear `passwordResetToken`/`passwordResetExpiresAt`. Return success or a clear "This reset link is invalid or has expired" error.

3. Frontend: a simple `/dealer/forgot-password` page (email input, submit) and `/dealer/reset-password` page (reads `?token=` from URL, shows a new-password input, submits). Match existing site branding (burgundy/gold/cream).

## Part 6: Shared dealer-overview component

Create `DealerTerritoryOverview.jsx` — a presentational component that takes props (dealer info, territories array with stats, totals, commission data) and renders:
- Dealer header (name, status, email, etc.)
- Overall stats cards (using a `commissionCents` figure — see Part 7 below — NOT raw revenue, when rendered for the dealer view; see prop below)
- Per-territory cards: city/territory name, landing page link + copy-link button, published status, fill bar (sold/total spots), ad-picker thumbnails (whatever you're adding there — wire this component to accept and render a `thumbnails` array per territory if available), households/ZIP count
- Accepts a boolean prop `showRevenue` — when true (admin view), show exact revenue dollar figures; when false (dealer view), show the commission figure instead (see Part 7) and omit raw revenue entirely

## Part 7: Commission calculation

Dealer commission per sold spot = **35% of the spot's sale price** (this is `(price / 2) * 0.70`, simplified to `price * 0.35` — confirm this matches: a $499 XL spot → $499 × 0.35 = $174.65 ✓; a fully-sold 15-spot campaign averaging ~$400/spot ≈ $6,000 total revenue → $6,000 × 0.35 = $2,100 ✓).

Add a shared constant `DEALER_COMMISSION_RATE = 0.35` in a small shared lib file (e.g. `lib/commission.ts`) — do NOT hardcode 0.35 inline in multiple places, since this rate may change again later (it was 80%→ now 70% of half, i.e. 40% → 35%, so it has changed before and will likely change again).

For any endpoint feeding the dealer-facing view, compute `commissionCents = Math.round(revenueCents * DEALER_COMMISSION_RATE)` per territory and as a total, alongside (not instead of) the existing `revenueCents` field in the API response — let the FRONTEND decide whether to show revenue or commission based on which view it is, rather than having two different backend responses. This keeps one source of truth for the calculation.

## Part 8: Two routes, one component

1. **Admin route** (existing, extend from earlier work): `/admin/dealers/:id` — fetches via `GET /api/admin/dealers/:id` (admin JWT), renders `<DealerTerritoryOverview ... showRevenue={true} />`.

2. **New dealer-facing route**: `/dealer/dashboard` — requires the dealer to be logged in (check for the dealer JWT in localStorage under e.g. `dealerToken`; if missing/invalid, redirect to a new `/dealer/login` page). Fetches via a new endpoint `GET /api/dealer/me` (protected by `requireDealerAuth`, uses `req.dealerId` from the verified JWT — a dealer can ONLY ever see their own data, there is no `:id` param on this route, it's always "me") which returns the same shape of data as the admin endpoint. Renders `<DealerTerritoryOverview ... showRevenue={false} />`.

3. Build a simple `/dealer/login` page: email + password fields, submit to `POST /api/dealer/login`, store the returned token in localStorage, redirect to `/dealer/dashboard`. Include a "Forgot password?" link to `/dealer/forgot-password`.

---

After implementing, test the full flow: create a dealer with a password via the admin tool, log in at `/dealer/login` with that email/password, confirm `/dealer/dashboard` shows their territories with commission figures (not raw revenue), confirm the admin view at `/admin/dealers/:id` still shows full revenue for the same dealer, and confirm the forgot-password flow sends a working reset email end to end.
