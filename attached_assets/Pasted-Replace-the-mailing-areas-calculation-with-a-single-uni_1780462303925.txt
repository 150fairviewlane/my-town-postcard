════════════════════════════════════════════════
THE CHANGE
════════════════════════════════════════════════

In GET /api/territories/:id/mailing-areas:

Replace ALL existing logic with this:

  1. Fetch the territory record by ID
     Get centroid_lat and centroid_lng

  2. Call findCandidateHubs(centroid_lat, centroid_lng)
     This function already exists in territoryBuilder.ts
     It finds qualifying commercial hub cities near a location

  3. Call selectBestHubs(candidates, centroid_lat, centroid_lng)
     This function already exists in territoryBuilder.ts
     It selects the best 4 hubs with good geographic spread

  4. Return the hub city names as mailing areas:
     [
       { name: "Dahlonega" },
       { name: "Cleveland" },
       { name: "Helen" },
       { name: "Cornelia" }
     ]

That is the entire implementation. No other logic needed.

The frontend displays city names with the existing footer:
"Each mailing area reaches 5,000 local
households via USPS EDDM."

SAME-STATE FILTER
════════════════════════════════════════════════

findCandidateHubs already filters to the same state as
the dealer ZIP. For territory mailing areas, filter to
the same state as the territory record (territory.state).

Pass the territory's state abbreviation to findCandidateHubs
so it doesn't return hub cities across state lines.
