---
name: PDF ad image storage pattern
description: Where finished ad images live and how to load them for PDF generation
---

Finished ad images (generated by Grok/AdGen) are stored as base64 data URLs inside the `templateData` text column as `templateData.finishedAdUrl` (e.g. `data:image/jpeg;base64,...`).

The `adFileUrl` column on `spots` is always NULL for these ads.

**Why:** The ad generator stores the rendered canvas as a data URL directly into templateData at generation time. adFileUrl is only set for manually-uploaded files (Cloudinary).

**How to apply:**
1. Always SELECT `templateData` alongside `adFileUrl` in any query that needs spot images.
2. Prefer `adFileUrl` first; fall back to `JSON.parse(templateData).finishedAdUrl`.
3. For data: URIs, decode base64 directly to Buffer — never pass to `fetch()`.

```typescript
function resolveImageUrl(dbSpot): string | null {
  if (dbSpot.adFileUrl) return dbSpot.adFileUrl;
  if (dbSpot.templateData) {
    const td = JSON.parse(dbSpot.templateData);
    if (typeof td.finishedAdUrl === 'string') return td.finishedAdUrl;
  }
  return null;
}

async function loadImageBuffer(url: string): Promise<Buffer | null> {
  if (url.startsWith('data:')) {
    return Buffer.from(url.slice(url.indexOf(',') + 1), 'base64');
  }
  const res = await fetch(url, { signal: AbortSignal.timeout(10_000) });
  return res.ok ? Buffer.from(await res.arrayBuffer()) : null;
}
```

**PDF delivery on iOS Safari:** Use `Content-Type: application/octet-stream` + `Content-Disposition: attachment` so iOS saves to Files app instead of trying to render inline (which shows a blank page for large images). Frontend uses `fetch()` → blob URL → `<a>.click()` instead of `window.open()`.
